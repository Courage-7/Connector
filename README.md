# Connector Service

Connector Service is a focused FastAPI microservice that lets a client application or AI agent:

- discover supported providers and their typed tools;
- send a user through delegated OAuth;
- persist encrypted provider connections in Supabase Postgres;
- execute owner-scoped tools through REST or Streamable HTTP MCP.

There is no frontend, project-key issuance flow, admin API, grant system, or approval engine in this
phase. The service uses one ordinary `Authorization: Bearer` scheme. Swagger UI is the development
and API-testing surface.

## Phase status

| Provider | Discovery | OAuth | REST tools | MCP tools |
| --- | --- | --- | --- | --- |
| Supabase | Ready | Ready | Ready | Ready |
| Google Workspace (Gmail + Calendar) | Ready | Next phase | Next phase | Next phase |
| Microsoft 365 (Outlook + Calendar + Teams) | Ready | Next phase | Next phase | Next phase |

The current Supabase tools list projects, select a project, list tables, describe a table, and run a
bounded structured query. Queries are generated by the service and execute through Supabase's
read-only Management API endpoint; callers cannot submit arbitrary SQL.

## Architecture

```text
Application / person / AI agent
        |
        | Authorization: Bearer <token>
        v
FastAPI REST + Swagger  ------------------  Streamable HTTP MCP (/mcp)
        |                                      |
        +---------- shared application tools --+
                           |
              owner-scoped connections
                           |
           encrypted tokens in Supabase Postgres
                           |
                  provider OAuth/API
```

The Bearer token identifies the service user. In local development it is a configured static token.
In production it is a Supabase Auth user access token verified locally against the project's JWKS.
Provider access and refresh tokens are encrypted before storage and never returned by the API.

See [provider architecture](docs/architecture/provider-modules.md) for package boundaries.

## Local setup

Requires Python 3.12+ and a dedicated Supabase project for connector metadata.

```powershell
python -m venv .venv
.venv\Scripts\Activate.ps1
python -m pip install -e ".[dev]"
Copy-Item .env.example .env
```

Generate the two local secrets and paste them into `.env`:

```powershell
python -c "import secrets; print(secrets.token_urlsafe(48))"
python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

Use the first value for `SERVICE_BEARER_TOKEN` and the second for
`TOKEN_ENCRYPTION_KEY`. Copy `DATABASE_URL` from the dedicated Supabase project's **Connect** dialog,
then add the Supabase OAuth client ID and secret. Detailed instructions for every value are in the
[environment setup guide](docs/configuration/environment-setup.md).

Start the service on its fixed development port:

```powershell
python -m uvicorn connector_service.main:app --reload --port 1080
```

Open:

- Swagger UI: <http://127.0.0.1:1080/docs>
- ReDoc: <http://127.0.0.1:1080/redoc>
- OpenAPI JSON: <http://127.0.0.1:1080/openapi.json>
- MCP endpoint: `http://127.0.0.1:1080/mcp`

## Try the Supabase flow in Swagger

1. Select **Authorize** and enter only the `SERVICE_BEARER_TOKEN` value. Swagger adds `Bearer`.
2. Execute `POST /v1/connections/supabase/authorize`.
3. Copy `authorization_url` from the response and open it in a browser. This explicit handoff is
   intentional: Swagger's background HTTP request cannot safely navigate the browser with a Bearer
   header.
4. Sign in to Supabase and approve the OAuth app. Supabase returns to the callback on port 1080.
5. Copy the returned `connection.id`.
6. Execute the Supabase tools in order: `list-projects`, `select-project`, then the database tools.

The complete request bodies and troubleshooting notes are in the
[Swagger guide](docs/api-and-swagger-guide.md).

## Use MCP

MCP runs in the same process and uses the same Bearer token and persisted connections as REST.
Configure a Streamable HTTP client with:

```text
URL: http://127.0.0.1:1080/mcp
Authorization: Bearer <SERVICE_BEARER_TOKEN>
```

`GET /v1/mcp` lists the currently exposed MCP tools. This is our stable service facade; provider-run
MCP servers are not used because they do not replace our cross-provider OAuth ownership,
persistence, common contracts, or per-user tenant boundary.

## Production authentication

Set:

```dotenv
ENVIRONMENT=production
AUTH_MODE=supabase_jwt
SUPABASE_AUTH_URL=https://YOUR_AUTH_PROJECT_REF.supabase.co
SUPABASE_AUTH_AUDIENCE=authenticated
AUTO_CREATE_SCHEMA=false
```

Run migrations during deployment:

```powershell
python -m alembic upgrade head
```

The consuming application signs users in with Supabase Auth and forwards each user's access token in the
standard `Authorization` header. Static authentication is rejected in production.

## Verification

```powershell
python -m pytest
python -m ruff check src migrations tests
python -m ruff format --check src migrations tests
python -m compileall -q src tests
```

The deterministic tests cover Bearer and JWT authentication, OAuth replay protection, encrypted
token persistence, connection ownership, restart persistence, the Supabase tool flow, Swagger, and
MCP authentication. Provider network calls are mocked; live credentials are never required by the
test suite.
